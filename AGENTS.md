# Роли агентов — FootyCompare

Дирижёр (главный цикл) оркеструет и НЕ пишет крупный код сам — делегирует специалистам, гоняет
гейты и ревью, коммитит+пушит. Независимые задачи одной фазы — параллельно.

| Агент | Когда | Что делает |
|---|---|---|
| **Architect** | Фаза 0 | Скаффолд, конфиги, дизайн-токены, i18n-каркас |
| **Data Engineer** | Фаза 1 | Схема, `DataSource`, парсеры (Wikidata/Understat + best-effort FBref/TM с фолбэком), ingestion, резалки-срезы, DATA_REPORT |
| **Card/Design Engineer** | Фаза 2 | Компонент карточки по эталону + рендер в PNG (Playwright). **Обязателен скилл `ui-ux-pro-max`** |
| **UI Engineer** | Фаза 3 | Страница, селекторы срезов, предпросмотр, скачать/поделиться, переключатель языка |
| **Design / Motion** | Фаза 4 | Анимации, адаптив, доводка визуала. **Обязателен `ui-ux-pro-max`** |
| **Tester** | После каждой задачи | Vitest + Playwright; ЗАПУСКАЕТ гейты `typecheck && lint && test && build`; PASS/FAIL с логами; НЕ «зеленит» силой |
| **Reviewer** | Перед коммитом | Чистота/переиспользование/баги (подход /simplify); без `any`, мёртвого кода, дублей |
| **Visual QA / Pixel-Critic** | Перед коммитом ЛЮБОЙ визуалки (в ДОПОЛНЕНИЕ к Tester) | Независимый adversarial-критик «как придирчивый человек», по умолчанию считает что что-то не так. Скриншотит КАЖДЫЙ экран на desktop 1440 И mobile истинный 390 (CDP), читает глазами + сверяет с `design-refs/`, выдаёт скрупулёзный список дефектов. PASS только при 0 дефектов; иначе FAIL → автор чинит → прогон СНОВА до PASS |

## Visual QA / Pixel-Critic — процесс (обязателен для любой визуальной задачи)
Отдельный агент, НИКОГДА не автор кода и отдельно от функционального Tester. Adversarial-настрой: ищет дефекты, а не подтверждает что «и так норм».
1. Поднять живой сайт; через Playwright снять скриншоты КАЖДОГО экрана (`/`, `/compare`, `/verdict`, `/cards`, шер-модалка) на **desktop 1440** И **mobile ИСТИННЫЙ 390** через `CDP Emulation.setDeviceMetricsOverride` (native window-size врёт ~500). Мотион ON, реальный `page.mouse.wheel` чтобы whileInView-ревилы сработали.
2. На каждый экран: прочитать скриншот ГЛАЗАМИ (vision) И соответствующий референс из `design-refs/` (ref1 арена, ref2 FUT, ref3 флоу/дизайн-система).
3. Скрупулёзный список дефектов: всё что наезжает/касается не так хоть на пиксель, любые съезды/несоосность, кривые отступы/сетки, не по центру, битые/уродские картинки, отклонения цвета/контраста/шрифта, и всё что выглядит ХУЖЕ или НЕ ПОХОЖЕ на референс. На каждый дефект: экран, элемент, что не так, серьёзность.
4. ВЕРДИКТ: PASS только если 0 дефектов; иначе FAIL со списком → автор чинит → прогнать СНОВА, и так пока не PASS.

## Правила
- В промт субагенту всегда: путь к `SPEC.md` + `AGENTS.md`, конкретная задача из `PLAN.md`, Definition of Done (SPEC §11).
- Tester и Reviewer — ВСЕГДА отдельные агенты от автора кода.
- FAIL у Tester → возврат автору с фидбэком, повтор, и только потом коммит.
- Задача `[x]` только когда все гейты зелёные + ревью пройдено.
- Любой UI/визуал → обязательно через скилл `ui-ux-pro-max` (и shadcn MCP при необходимости).
- Любой UI/визуал → ОБЯЗАТЕЛЕН прогон **Visual QA / Pixel-Critic** до PASS (0 дефектов) ПЕРЕД коммитом, в дополнение к Tester.
