# FootyCompare — ТЗ (источник правды) · MVP

> Сервис-генератор красивых карточек-сравнений **Месси vs Роналду** по любому сезону/контексту.
> Это файл-источник правды для дирижёра и всех агентов. Читать ПЕРЕД каждой итерацией.

---

## 1. Продукт (одно предложение)

**FootyCompare** — веб-сервис, где выбираешь параметры/период и получаешь **красивую карточку-сравнение Месси и Роналду**, готовую к шерингу в соцсети. Карточка нейтральна по фактам, но показывает «счёт по категориям» — виральный крючок для спора.

- **Для кого:** футбольные болельщики, спорщики, контент-мейкеры.
- **Зачем шарят:** карточка = патрон в вечном споре «Месси или Роналду». Человек репостит её со своей подписью.
- **Moat:** (1) самая красивая карточка на рынке, (2) минимум трения при шеринге.

## 2. MVP-скоуп

**Входит:**
- Только **два игрока: Месси и Роналду** (хардкод).
- Генерация **статичной карточки-картинки** (PNG), формат вертикальный ~2:3 (под Stories/Telegram/X).
- Срезы (см. §5): выбор сезона каждому · фильтр по турниру · сравнение в одном возрасте · пенальти вкл/выкл.
- «Счёт по категориям» (напр. 6:3) — механический подсчёт, кто ведёт по скольким параметрам.
- Скачать/поделиться карточкой.
- **Реальный ingestion-pipeline с парсерами** (см. §6) — данные парсятся сразу, это часть архитектуры. Помечены `verified:false` → before-launch сверка.

**НЕ входит (явно отложено):**
- Другие игроки (расширение — потом, в ту же машину).
- Видео-карточка и авто-постинг в TikTok (это уровень приложения, не MVP-сайта).
- Кастомный конструктор тем (на старте — 1 вылизанный шаблон).
- Ручная экспертная сверка каждой цифры (это шаг владельца после сборки, по отчёту).

## 3. Ядро: вердикт и карточка

- **Вердикт = «счёт по категориям».** По выбранному набору статов считаем, по скольким ведёт каждый → «MESSI 6 : 3 RONALDO» + «6 categories won». Формулировка строго механическая: **никогда не «X лучше»**, только «ведёт по N параметрам».
- **Единица продукта — не страница, а картинка.** Весь UX подчинён тому, чтобы за пару тапов получить и расшарить красивую карточку. Если человек «завис в интерфейсе» — это провал.
- На карточке всегда **вотермарк/лого FootyCompare** (бесплатный маркетинг в каждом репосте).

## 4. Эталон карточки

Ориентир — мокап от ChatGPT (тёмная тема, два игрока, «VS», селектор периода, список статов с двойными барами розовый/синий, внизу «OVERALL RESULT» + счёт, снизу атрибуция источников). Элементы карточки:
- **Шапка:** имя+фамилия, фото игрока, клуб (для выбранного сезона), позиция, флаг сборной; по центру «VS».
- **Плашка периода/контекста:** «PERIOD: 2011/12» или «LAST 5 YEARS», + активные фильтры (турнир, возраст, пенальти).
- **Список статов:** на каждый — двойной бар (Месси розовый `#E91E8C`-зона / Роналду синий `#2EA8FF`-зона) + значения с обеих сторон, иконка параметра. Лучшее значение визуально выделено.
- **Итог:** «OVERALL RESULT — счёт N:M, categories won».
- **Футер:** вотермарк FootyCompare + (позже) атрибуция источников данных.
- Дизайн уровня бога: glassmorphism + неон, аккуратная типографика, `tabular-nums`, идеальные отступы. Делать через скилл **`ui-ux-pro-max`**.

## 5. Срезы (must-have для MVP)

Все управляют тем, какие строки данных суммируются:
1. **Выбор сезона каждому** — Месси 11/12 vs Роналду 14/15 (или «вся карьера» / «последние 5 лет»).
2. **Фильтр по турниру** — Всё / Лига / Лига Чемпионов / Сборная.
3. **Сравнение в одном возрасте** — «Месси в 25 vs Роналду в 25» (выравнивание по возрасту, а не по календарю).
4. **Пенальти вкл/выкл** — голы с учётом пенальти или без.

## 6. Модель данных (КРИТИЧНО — несущая стена)

Атомарная единица — **одна строка на (игрок × сезон × турнир)**. Из кирпичиков вычисляется ЛЮБОЙ срез (сумма/фильтр). НЕ хранить готовые career-тоталы.

```ts
type PlayerSeasonComp = {
  player: "messi" | "ronaldo";
  season: string;            // "2011/12"
  ageDuringSeason: number;   // для среза "в одном возрасте"
  club: string;              // клуб в этом сезоне
  competitionType: "league" | "champions_league" | "domestic_cup"
                 | "super_cup" | "club_world_cup" | "national_team";
  competitionName: string;   // "La Liga", "UEFA Champions League", ...
  // игровое время
  matches: number; starts: number; minutes: number;
  // голы (с разбивкой — нужно для среза пенальти)
  goals: number; penaltyGoals: number; freekickGoals: number;
  // созидание
  assists: number;
  // удары
  shots: number; shotsOnTarget: number;
  // продвинутое (ТОЛЬКО для сезонов ~2014+, иначе null)
  xg: number | null; xa: number | null;
  // дисциплина
  yellowCards: number; redCards: number;
  // достижения сезона
  trophies: string[];        // ["La Liga","Copa del Rey"]
  individualAwards: string[];// ["Ballon d'Or","European Golden Shoe"]
};
```

- **Реальный ingestion-pipeline (РЕШЕНО), стратегия источников — ГИБРИД:**
  - **Основа (надёжная, легальная):** Wikidata/Wikipedia (API/SPARQL — сезоны, голы, ассисты, матчи, трофеи, награды, возраст) + Understat (xG/xA для сезонов 2014+).
  - **Обогащение (по возможности, не блокер):** попытка FBref/Transfermarkt для деталей; если бан/анти-бот/недоступно — **деградация на правдоподобный seed**, помечается в отчёте, сборка продолжается.
  - Парсеры пишут в каноническую схему `PlayerSeasonComp`, каждый источник за адаптером.
- **`verified: false` на всех записях** + before-launch отчёт `DATA_REPORT.md`: что откуда взято, что с фолбэка, что требует ручной сверки.
- Покрыть всю карьеру: Месси (с 2004/05) и Роналду (с 2002/03), разбивку по турнирам, возраст по сезонам, пенальти — чтобы все 4 среза реально работали.
- Слой данных за интерфейсом `DataSource` — источники взаимозаменяемы **без переписывания фронта**.
- Парсинг — это **разовый ingestion в локальный кэш/JSON** (не дёргаем внешние сайты на каждый запрос пользователя): спарсили → нормализовали → сохранили → фронт читает из своей базы.
- **Производные метрики** (конверсия, голы/90, xG/90) считаются из строк, не хранятся.
- Граница честности: продвинутые метрики (xG/xA) — `null` для сезонов до ~2014, и в UI для тех лет просто не показываются.

## 7. Набор статов на карточке (~10-12)

Голы · Ассисты · Матчи · Минуты · Голы/90 · Конверсия ударов · xG (совр.) · xA (совр.) · Трофеи (кол-во) · Золотые мячи · Хет-трики (если есть в данных) · Красные/жёлтые (опц.).
Каждый стат — **один канон-источник + одно определение**, подписанное (важно для будущих реальных данных; в mock просто закладываем поля).

## 8. Фото игроков

- MVP: **2 реальных фото** (Месси, Роналду) — кладём как ассеты, слот в карточке.
- Механика: компонент фото принимает src; легко заменить.
- ⚠️ **Before-launch TODO (не блокер MVP, но записать в отчёт):** оформить права/лицензию на изображения ИЛИ заменить на стилизованный арт — реальные фото на виральном масштабе = копирайт/право на образ. На этапе MVP допустимо, на паблик-запуске — решить.

## 9. Шеринг / экспорт

- Кнопка **«Скачать PNG»** (главное для MVP).
- Кнопка **«Поделиться»** через системный Web Share API (share-sheet на мобиле).
- Карточка генерится как изображение на сервере (см. §10).

## 10. Стек и техника

- **Next.js 15 (App Router) + React 19 + TypeScript strict + Tailwind v4** (как и раньше).
- **Генерация карточки в изображение — РЕШЕНО: headless-браузер (Playwright screenshot).** Рендерим карточку как обычную страницу (полный Tailwind/CSS: glassmorphism, backdrop-blur, тени, градиенты, неон), делаем скриншот нужного вьюпорта → PNG. Это даёт максимум визуала (Satori умышленно НЕ берём — у него ограничения CSS, а красота карточки = главный moat). Эндпоинт `/api/card` рендерит и отдаёт PNG.
- **Формат карточки — РЕШЕНО: вертикаль ~2:3** (напр. 1080×1620), под Stories/Telegram/TikTok-превью.
- **i18n — РЕШЕНО: двуязычно (RU + EN), переключатель.** Все подписи на карточке и в UI — через словарь, дефолт можно EN. Числа/иконки общие.
- **Темы — РЕШЕНО: один вылизанный тёмный-неон шаблон** (как эталон §4). Пресеты — потом.
- `framer-motion` для UI (не для самой картинки), `lucide-react` иконки.
- **pnpm**. Алиас `@/`. Playwright уже нужен и для рендера, и для e2e.
- Качество кода: strict, без `any`, ESLint+Prettier зелёные, маленькие чистые компоненты.

## 11. Фазы (план для дирижёра) + Definition of Done

Каждая фаза — законченная, проверяемая, заканчивается коммитом+пушем. **Между фазами — контрольная точка для владельца** (особенно Фаза 2 и Фаза 4).

- **Фаза 0 — Каркас:** Next.js+TS+Tailwind, тулинг (ESLint/Prettier/Vitest/Playwright), дизайн-токены (через `ui-ux-pro-max`), CI-гейты зелёные.
- **Фаза 1 — Данные (ingestion-pipeline):** схема `PlayerSeasonComp`, `DataSource`-интерфейс; **парсеры-адаптеры** (Wikidata/Wikipedia + Understat — основа; FBref/Transfermarkt — обогащение с фолбэком на seed); разовый ingestion → нормализация → сохранение в локальную базу/JSON; функции-резалки (4 среза → агрегаты); `verified:false` + `DATA_REPORT.md`; юнит-тесты нормализации и агрегации (на фикстурах, без живой сети).
- **Фаза 2 — Генератор карточки:** компонент карточки по эталону §4 + рендер в PNG (Satori/Playwright), эндпоинт `/api/card`. ⚠️ **Контрольная точка: владелец смотрит карточку глазами.**
- **Фаза 3 — Интерфейс:** страница с селекторами всех 4 срезов (§5) → живой предпросмотр карточки → кнопки «Скачать/Поделиться».
- **Фаза 4 — Полировка:** анимации, адаптив, скелетоны, идеальный визуал карточки (через `ui-ux-pro-max`). ⚠️ **Контрольная точка: владелец.**
- **Фаза 5 — Приёмка:** Playwright E2E happy-path, `/code-review`, BUILD_REPORT.md (что собрано, как запустить, превью карточки, **список before-launch TODO: данные на сверку + лицензия фото**), стоп.

**Definition of Done (каждая задача):** `pnpm typecheck` ✓ · `pnpm lint` ✓ · `pnpm test` ✓ · `pnpm build` ✓ · (для экранов) Playwright-smoke ✓ · ревью отдельным агентом ✓ · отметка в PLAN.md + коммит (Conventional Commits).

## 12. Предохранители автономного запуска (из ночных шишек)

- **Сначала** подними тулчейн (node 20 + pnpm) и настрой git-push (надёжно), потом сборку.
- **Коммит+пуш после каждой фазы** — чтобы падение/перезапуск не терял прогресс.
- **Луп — короткий интервал/подряд**, не раз в час.
- Если процессы внезапно умирают (OOM/SIGKILL) — **проверь `/proc` на жадные чужие процессы** перед тем как винить код.
- В конце — **артефакты на сверку владельцу**: превью карточки + отчёт по данным (что mock, что надо выверить).

## 13. Роли агентов

Дирижёр оркеструет, не пишет крупный код сам. Специалисты: Architect (каркас), Data Engineer (схема+mock+агрегации), Card/Design Engineer (карточка+рендер, **обязателен `ui-ux-pro-max`**), UI Engineer (селекторы+страница), **отдельный Tester** (гейты, не «зеленит» силой), **отдельный Reviewer** (чистота/баги). Tester и Reviewer — всегда отдельные от автора.
